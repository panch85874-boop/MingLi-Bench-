## 1. 總評：架構是否可行

> **結論：方向正確、可行，MVP 順序與收費時機都站得住腳。主要待補的是「併發與容錯的工程細節」與「退款配套」，而非架構本身要重來。**

原稿做對的關鍵決策（建議維持，不要在後續被推翻）：

| 決策 | 評價 | 理由 |
|---|---|---|
| 四層分離（掃描設備 / 編排 / OCR 工廠 / 顧客端） | ✅ 正確 | 責任邊界乾淨，可獨立替換、獨立擴充 |
| C3570 當 dumb device，只做紙本→PDF→FTP | ✅ 正確 | 換廠牌、機器故障時上層不受影響 |
| OCR/Agent 設計成無狀態 worker | ✅ 正確 | 可中央化、可水平擴充、可排隊 |
| 先掃描 → 算頁數 → 報價 → 付款 → 才 OCR | ✅ 正確 | 頁數與品質掃完才知道，先付款必生退款 |
| MVP：可搜尋 PDF → Word → Excel | ✅ 正確 | 難度與風險遞增，營收先跑、難題後攻 |
| OCR 中央部署而非每店 GPU | ✅ 正確 | 算力集中攤提、統一升級、共用排隊 |

需要補強的核心缺口（本文件主要貢獻）：

1. **裝置鎖 / 掃描檔對應任務**（第 4 章）— 最容易被低估、最早出事的併發 bug 源。
2. **FTP 上傳完整性偵測**（第 3 章）— Scan to FTP 不送「完成」訊號。
3. **退款流程**（第 6 章）— 「先收費後 OCR」的必然配套，非 optional。
4. **狀態機補轉移**（第 2 章）— 逾時、退款、重試三條路徑。
5. **付款前預覽 + 免責確認**（第 7 章）— 壓低退款與客訴的最高 CP 值改動。

## 2. 強化版任務狀態機

### 2.1 原稿狀態 + 新增狀態

原稿狀態保留，新增 3 個狀態以涵蓋逾時、退款、重試：

```text
CREATED
WAITING_SCAN
FILE_RECEIVED
PRICED
WAITING_PAYMENT
PAID
OCR_QUEUED
OCR_PROCESSING
EXPORTING
COMPLETED
FAILED
EXPIRED
CANCELLED
# ── 新增 ──
REFUNDING        付款後無法產出結果，退款處理中
REFUNDED         已退款完成
RETRYING         OCR 失敗，自動重試一次中
```

### 2.2 完整狀態轉移圖

```text
                         ┌──────────┐
                         │ CREATED  │
                         └────┬─────┘
                              ↓
                     ┌────────────────┐
                     │  WAITING_SCAN  │──── 10 分鐘未收到 PDF ──→ EXPIRED
                     └────────┬───────┘
                              ↓ （FTP 收到並確認完整）
                     ┌────────────────┐
                     │ FILE_RECEIVED  │──── 品質過差/頁數超限 ──→（顧客確認是否繼續）
                     └────────┬───────┘                              │
                              ↓                                       ↓ 放棄
                       ┌──────────┐                              CANCELLED
                       │  PRICED  │
                       └────┬─────┘
                            ↓
                   ┌─────────────────┐
                   │ WAITING_PAYMENT │── 報價後 N 分鐘未付款 ──→ EXPIRED
                   └────────┬────────┘
                            ↓ （Store Host poll 到付款成功）
                       ┌─────────┐
                       │  PAID   │
                       └────┬────┘
                            ↓
                     ┌─────────────┐
                     │ OCR_QUEUED  │
                     └──────┬──────┘
                            ↓
                  ┌──────────────────┐
                  │  OCR_PROCESSING  │
                  └────────┬─────────┘
                           ↓
                     ┌───────────┐         OCR 失敗（第 1 次）
                     │ EXPORTING │──────────────┐
                     └─────┬─────┘              ↓
                           ↓                ┌──────────┐
                     ┌───────────┐          │ RETRYING │── 成功 ──→ EXPORTING
                     │ COMPLETED │          └────┬─────┘
                     └─────┬─────┘               │ 第 2 次仍失敗
                           ↓                      ↓
              下載/Email 後依政策清理        ┌────────┐
                           ↓                 │ FAILED │
                        EXPIRED              └───┬────┘
                  （結果檔逾時刪除）              ↓ 已付款
                                            ┌───────────┐
                                            │ REFUNDING │
                                            └─────┬─────┘
                                                  ↓
                                            ┌──────────┐
                                            │ REFUNDED │
                                            └──────────┘
```

### 2.3 新增/明確化的轉移規則

| 轉移 | 觸發條件 | 處理 |
|---|---|---|
| `WAITING_SCAN → EXPIRED` | 任務建立後 10 分鐘未收到 PDF | 釋放裝置鎖、提示重新開始 |
| `WAITING_PAYMENT → EXPIRED` | 報價後 N 分鐘（建議 15 分）未付款 | 釋放裝置鎖、刪除暫存 PDF |
| `EXPORTING → RETRYING` | OCR/匯出第 1 次失敗 | 自動重試一次（同一份輸入） |
| `RETRYING → FAILED` | 重試後仍失敗 | 進入退款判斷 |
| `FAILED → REFUNDING` | 失敗且 `status` 曾達 `PAID` | 觸發退款（見第 6 章） |
| `REFUNDING → REFUNDED` | 金流退款確認成功 | 通知顧客、記錄對帳 |
| `FILE_RECEIVED → CANCELLED` | 品質警示後顧客選擇放棄 | 未收費，直接釋放鎖與暫存 |

> **重點：`FAILED` 必須區分「付款前失敗」與「付款後失敗」。** 付款前失敗（如掃描品質爛）直接結束即可；付款後失敗一定要進 `REFUNDING`，否則就是「收了錢沒交付」的客訴與商譽風險。

## 3. FTP 上傳完整性偵測（被低估的技術風險 #1）

### 3.1 問題

Fuji C3570 的 **Scan to FTP 不會送出「我傳完了」的訊號**。Store Host 監看 `scan-inbox` 時，看到一個檔案出現，**不代表它已經傳完**。多頁掃描尤其明顯：檔案會逐漸長大。若此時就搬移、計頁數、報價，會抓到**半截 PDF**，導致：

- 頁數算錯 → 報價錯誤
- PDF 損壞 → OCR 失敗（但顧客已被報價甚至付款）
- 搬移到一半的檔案 → 檔案鎖定衝突

### 3.2 建議偵測機制（三選一或併用）

**方案 A：檔案大小穩定偵測（最通用，建議預設）**
```text
偵測到新檔 → 每 2 秒檢查一次檔案大小
→ 連續 3 次（約 6 秒）大小不變 且 可取得獨占讀取鎖
→ 視為上傳完成 → 才搬移
```

**方案 B：暫存檔改名（若 C3570/FTP 支援）**
```text
掃描中寫入 scanfile.pdf.part
→ 完成後 rename 為 scanfile.pdf
→ Store Host 只認 *.pdf，忽略 *.part
```

**方案 C：FTP server 端 hook（若用自建 FTP 服務）**
```text
FTP STOR 完成事件 → 觸發通知 Store Host
```

### 3.3 額外保護

- 搬移前用 PDF 解析庫**驗證檔案結構完整**（能正確開啟、頁數可讀），失敗則標記「疑似上傳未完成」並重試讀取，而非直接報價。
- 設定**單檔最大等待時間**（如 90 秒大小仍在變動 → 視為異常大檔或卡住，警示後台）。

> 負責邊界：此項完全屬於 **Store Host**，C3570 與 OCR 不參與。

## 4. 裝置鎖與「掃描檔 ↔ 任務」綁定（被低估的技術風險 #2）

### 4.1 問題

原稿用「`device-A` 對應一個 `WAITING_SCAN` 任務」**隱含**地解決綁定，但沒有處理併發：

- 顧客 A 在手機建任務（device-A，等待掃描），人還沒走到機器
- 顧客 B 走到同一台 device-A，也想用，建了另一個任務
- inbox 進來一個 PDF → **到底算 A 的還是 B 的？**

或者：
- 顧客 A 一次掃兩疊（中間分兩次按掃描）→ 進來兩個 PDF，哪個對應任務？

這是**最容易在實機測試才爆、且難重現**的 bug。

### 4.2 建議：裝置鎖（device lock）

```text
規則 1：同一個 device，同一時間只允許一個「待掃描中」的任務。
        建任務時嘗試取得 device lock；取不到 → 提示「此設備使用中，請稍候」。

規則 2：lock 有 TTL（= 掃描逾時，建議 10 分鐘），逾時自動釋放。

規則 3：任務進入 FILE_RECEIVED / EXPIRED / CANCELLED 時立即釋放 lock。
```

### 4.3 掃描檔對應任務的判定

在「同裝置同時只有一個待掃任務」的前提下，綁定就單純了：

```text
inbox 進來的 PDF（已確認上傳完整）
→ 找出該 device 目前唯一的 WAITING_SCAN 任務
→ 綁定該 job_id
→ 若該 device 沒有任何 WAITING_SCAN 任務（例如已逾時）
   → PDF 進「孤兒檔」暫存區，警示後台，N 分鐘後刪除
```

對於「一個任務、多次掃描產生多個 PDF」：
```text
建議第一階段：一個任務 = 一個 PDF。
顧客需多疊文件 → 引導一次放入或分次建立多個任務。
若要支援多檔合併，需明確「掃描結束」訊號（例如手機按「我掃好了」），
不要靠 timeout 猜，否則頁數與報價不穩。
```

### 4.4 邊界情況清單（請在規格中逐一定義行為）

| 情況 | 建議行為 |
|---|---|
| 任務逾時後 PDF 才到 | 進孤兒檔區，不綁定，警示後台 |
| 同任務進來 2 個 PDF | 第一階段：只取第一個，第二個進孤兒區並提示 |
| PDF 到了但找不到對應任務 | 孤兒區暫存 + 後台告警，N 分鐘自動刪 |
| 顧客建任務後從未去掃 | 10 分鐘 EXPIRED，釋放鎖 |
| 同顧客連續建多個任務 | 限制每裝置/每手機同時進行中任務數（建議 1） |

## 5. Store Host 狀態持久化與單點風險

### 5.1 問題

Store Host 同時是：流程中心 + 檔案中心 + 金流 poll + 清理排程。它**重啟一次**，記憶體中的進行中任務就消失，會造成：

- 已付款但 OCR 還沒派 → 顧客付了錢任務卻不見
- 金流 poll 中斷 → 付款成功但系統不知道
- 清理排程遺失 → 暫存檔永久殘留（資安風險）

### 5.2 建議

1. **所有任務狀態持久化到本機 DB**（SQLite/嵌入式即可，單店規模足夠），不可只存記憶體。
2. **Store Host 啟動時做 recovery 掃描**：
   - 載入所有未終結任務（非 COMPLETED/REFUNDED/EXPIRED/CANCELLED）
   - 重建金流 poll、OCR 派工、清理排程
3. **金流 poll 設計成可恢復**：以 `job_id` 對應交易單號，重啟後能重新查詢。
4. **清理工作以 DB 中的「保存到期時間」為準**，而非記憶體計時器，確保重啟後仍會清。

> 單店架構下，Store Host 是單點故障無法完全避免，但「持久化 + 啟動復原」能把「重啟 = 資料遺失」降為「重啟 = 短暫中斷後自動續跑」。

## 6. 退款流程（先收費後 OCR 的必然配套）

### 6.1 為什麼一定要做

收費發生在 OCR **之前**，所以「已扣款 + 無法產出可用結果」是**必然會發生**的狀態，不是極端例外：

- OCR 服務當機 / 逾時
- 中央 OCR 佇列塞爆，超過可接受等待
- PDF 雖通過初步檢查，但 OCR 仍無法辨識
- 匯出 Word/Excel 階段失敗

沒有退款流程 = 每一次失敗都變成人工客服 + 商譽損傷。

### 6.2 建議退款流程

```text
EXPORTING 失敗（第 1 次）
   ↓
RETRYING（自動重試一次，同一份輸入，不再收費）
   ↓ 仍失敗
FAILED
   ↓ 檢查：此任務是否曾達 PAID？
   ├─ 否（付款前失敗）→ 直接結束，提示重新掃描，不涉退款
   └─ 是（付款後失敗）→ REFUNDING
                          ↓
                    呼叫金流退款 / 標記待退款
                          ↓ 金流確認
                       REFUNDED
                          ↓
              手機通知：「轉檔失敗，款項已退回」+ 後台對帳紀錄
```

### 6.3 退款政策需先決定的事項

| 問題 | 建議預設 |
|---|---|
| 自動退款還是人工審核退款？ | 第一階段：失敗自動進「待退款」清單，由店家/後台一鍵確認；金額小可全自動 |
| 部分成功（10 頁成功 8 頁）算不算交付？ | 第一階段不做部分退款，整單失敗才退，避免判定複雜 |
| 顧客「結果不滿意」可否退？ | 明確「AI 轉檔結果不保證、不因品質主觀不滿退款」，並在付款前免責呈現（見第 7 章） |
| 退款時效 | 依金流商規範，記錄交易單號以利對帳 |
| 重試是否重新收費 | 否，系統自動重試一次不收費 |

> 法律上要注意：「結果不滿意可否退」與「技術失敗退款」是兩件事，前者靠**付款前免責 + 預覽**處理，後者靠**自動退款**處理。混在一起會造成客服判斷困難。

## 7. 預覽與付款前確認機制

### 7.1 兩種預覽，目的不同

**(A) 付款前預覽（高優先，強烈建議）**
```text
掃描完成 → 手機顯示：
- 總頁數
- 首頁（或前幾頁）縮圖
- 報價
- 免責與品質提示（需顧客點「我已確認」才能付款）
```
目的：讓顧客在**付錢前**確認「掃對了、沒掃歪、沒空白、頁數正確」，
這是把退款與客訴壓到最低的**最高 CP 值改動**。

**(B) 取檔前結果預覽（次優先）**
```text
轉檔完成 → 先給低解析度 / 浮水印預覽
→ 顧客確認結果可用 → 下載完整檔需已付款
```
目的：降低「下載後才發現不能用」的爭議。第一階段可只做付款前預覽，
結果預覽列為第二階段。

### 7.2 付款前品質檢查（與預覽搭配）

Store Host 在報價前做**輕量**初步檢查（不需 OCR，純影像/PDF 層級）：

| 檢查項 | 動作 |
|---|---|
| 空白頁比例過高 | 提示「偵測到多張空白頁，是否繼續？」 |
| 解析度過低 | 提示「掃描解析度偏低，可能影響結果」 |
| 頁面嚴重歪斜 | 提示「文件可能歪斜」 |
| 頁數超過上限 | 阻擋或改用大檔計價 |
| PDF 結構損壞 | 視為上傳未完成，重讀或要求重掃 |

> 這些檢查的目的不是擋住顧客，而是**讓顧客在付款前知情並自行決定**，把責任明確化、降低事後客訴。

## 8. 資安與隱私強化

原稿第 12 章方向正確，補充落地細節：

| 原則 | 落地建議 |
|---|---|
| 不永久保存文件 | 以 DB 中「到期時間」驅動刪除，重啟後仍生效；定期掃描殘留檔 |
| 傳輸只在受控範圍 | Store Host ↔ 中央 OCR 走加密通道（TLS/VPN），不走明文 FTP/HTTP |
| 下載連結短效 | 連結帶簽章 + 短 TTL（如 30 分），逾時即失效；不可用可猜測的流水號 |
| 店員/他人看不到內容 | 後台預設只顯示任務狀態，文件內容存取需獨立授權與稽核紀錄 |
| Email 寄送後清理 | 寄出確認後刪暫存；附件或短效連結二擇一並明確政策 |
| 孤兒檔/失敗檔 | 一併納入清理排程，不可遺漏（這類最容易殘留） |
| 中央 OCR 多店共用 | 確保各店資料邏輯隔離，OCR 不留存輸入/輸出 |

> 特別提醒：**中央化 OCR 後，文件會離開店內**。隱私聲明與顧客同意條款需據此調整（原本「只在店內」的說法在中央化後不成立）。

## 9. 潛在風險總表

風險分級：機率（高/中/低）× 衝擊（高/中/低）。**P0 = 上線前必處理，P1 = 第一階段內處理，P2 = 觀察。**

### 9.1 技術風險

| # | 風險 | 機率 | 衝擊 | 等級 | 緩解 |
|---|---|---|---|---|---|
| T1 | FTP 半截檔被當完整檔（頁數/報價錯、OCR 失敗） | 高 | 高 | **P0** | 第 3 章完整性偵測 + PDF 結構驗證 |

## 12. 結語

原架構稿的**骨架是健康的**：分工乾淨、MVP 順序務實、收費時機正確。本審查補的不是「推翻重來」，而是**把架構從「示意圖」推進到「能扛真實併發與失敗」**所需的工程細節：

1. 三個 **P0 工程缺口**：FTP 完整性、裝置鎖、狀態持久化。
2. 一個 **P0 營運缺口**：退款流程（先收費後 OCR 的必然配套）。
3. 一個 **P0 體驗缺口**：付款前免責 + 預覽。
4. 一個 **P0 法務缺口**：中央化後文件離店，隱私條款需更新。

把第 10 章的 P0 檢查表做完，這項服務就具備上線的基本健全度。建議第一階段商品聚焦 **AI 掃描轉可搜尋 PDF + Word**，Excel 維持實驗性，按原稿節奏推進。
